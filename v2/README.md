# ETF-Master: 跨市場數據分析工具 (v1.0 & v2.0)

本專案是一個專為投資者設計的輕量級 CLI 工具，旨在統一台股（.TW）與美股 ETF 的數據存取介面，提供即時的折溢價診斷與成交量分析。

## 1. 專案簡介

### v1.0 設計動機
在開發 v1.0 時，核心痛點在於跨市場數據的不一致性（例如台股淨值抓取困難、美股與台股代碼格式不同）。 
- v1.0 成功實作了基礎的 `info`、`pd` 與 `vol` 指令，並透過 `Fetcher` 模組解決了多重數據源校驗問題。

### 從 v1.0 到 v2.0 的演化摘要
v2.0 在 v1.0 的穩定基礎上，針對「數據深度」與「決策效率」進行了大幅擴充，將工具從單點查詢提升為系統化掃描：

* **指令功能擴充**：新增 `div` (配息查詢) 指令，支援自動觀測配息頻率（月配/季配/年配）。
* **Watchlist 模式正式化**：支援讀取 `watchlist.txt`，並強化了對註解（#）與空白行的過濾邏輯，實現一鍵批次掃描。
* **本地快取與歷史回溯**：新增 `history.json` 自動紀錄每一次查詢快照，每檔標的自動保留最近 20 筆紀錄，方便觀察價格與折溢價趨勢。
* **🌟 亮點：多標的橫向比較模式**：
    * **視覺優化**：當輸入多個代碼或使用 `watchlist` 時，系統由垂直報告自動切換為**橫向表格 (Comparison Mode)**。
    * **智能排序**：系統會根據核心指標（如折溢價程度、量能爆發比）進行**自動排序**，讓最值得關注的標的一目瞭然。
    * **專業評估標籤**：渲染器會自動標記出「✨ 最具安全邊際」或「🏆 今日人氣王」等診斷資訊，直接輔助投資決策。
    * 可以想像 v1.0 你要看 10 檔 ETF 的折溢價，你需要向上滾動螢幕 10 次去對比；但在 v2.0，這會被排成一張整齊的表，**折價最多的會自動排在最上面**。

## 2. v1.0 設計決策 (Design Decisions)

在設計 v1.0 時，我採取了「低耦合、高預留」的策略，為 v2.0 的功能擴充鋪平了道路：

* **解耦 Fetcher 與 Validator**：
    * **決策**：API 原始數據必須先經過 `Fetcher` 封裝成 `ETFData` 物件，再由 `Validator` 執行校驗。
    * **預判邏輯**：未來若更換數據源，只需修改 `Fetcher` 內部實作，不影響上層驗證邏輯。
* **資料模型 (Data Model) 的欄位預留**：
    * **決策**：在 v1.0 的 `models.py` 中即預留了 `tr_annual_yield`、`latest_volume` 等欄位，即使當時的功能尚未完全使用。
    * **預判邏輯**：這避免了升級 v2.0 時需要大幅修改資料結構，確保了股息與成交量分析功能的順利接入。
* **指令介面的彈性設計**：
    * **決策**：在 v1.0 階段即配置 `nargs="+"` 支援多代碼輸入。
    * **演化**：在 v1.0 時，多標的輸入僅是多次重複輸出單一報告；這項預留設計讓 v2.0 能在不更動使用者指令習慣的前提下，直接升級為**橫向比較模式**，實現無縫的體驗升級。
* **資料驗證的嚴格性分級 (Validation Tiering)**：
    * **決策**：在 `validator.py` 中，針對不同指令實施不同等級的攔截策略。例如 `info` 指令對殖利率採用「寬鬆驗證」（缺失時補 0.0），而 `pd` 指令對淨值則採「嚴格驗證」。
    * **理由**：這體現了對資料「必要性」的判斷。殖利率屬於參考資訊，不應因缺失而中斷掃描；但淨值是計算折溢價的核心，缺失會導致投資誤導，故必須強制攔截。
* **渲染層的職責分離 (SRP in Renderer)**：
    * **決策**：將顯示邏輯獨立於 Engine 之外，並在 `v2/core/renderer.py` 中大量運用 `@classmethod` 與 `@staticmethod` 來區分邏輯。
    * **理由**：將「狀態診斷邏輯」（如判斷顯著折價）與「視覺輸出邏輯」（如表格渲染）分離，符合**單一職責原則 (Single Responsibility Principle)**。這不僅讓程式碼更整潔，也讓診斷標籤能在「單一模式」與「比較模式」之間高度複用。

## 3. v2.0 實作說明

### 3.1 需求映射與架構對應
我將 v2.0 的新需求精確映射至既有的模組化架構中，確保功能擴充不破壞原有的穩定性：
* **數據層 (Fetcher)**：實作 `fetch_dividends` 方法，透過 `yfinance` 獲取歷史配息數據。
* **持久化層 (Storage)**：新增 `storage.py` (HistoryManager)，專責處理 JSON 格式的歷史紀錄讀寫。
* **渲染層 (Renderer)**：於 `ETFRenderer` 內新增 `comparison` 模式，並透過狀態診斷邏輯提供投資建議。

### 3.2 關鍵實作選擇
* **快取自動管理機制**：
    * 在 `HistoryManager.save_record` 中設定 `MAX_RECORDS_PER_TICKER = 20`。
    * **考量**：確保 `history.json` 不會因長期頻繁查詢而導致檔案體積過大，在提供歷史回溯功能的同時兼顧磁碟空間與讀取效能。
* **Smart Yield 智能殖利率計算**：
    * 針對部分標的（如台股 ETF）遺失 `trailingAnnualDividendYield` 欄位的問題，實作了「手動回溯 365 天歷史股息加總」的邏輯。
    * **考量**：此邏輯能大幅提高跨市場數據的準確度，確保美股與台股在同一介面下擁有對等的評估基準。

### 3.3 錯誤處理與 UX 哲學
* **非阻斷式錯誤處理 (Non-blocking UX)**：
    * **策略**：在實作 `watchlist` 批次掃描或多標的查詢時，採用「單一失敗不中斷全域」的錯誤攔截邏輯。
    * **實作細節**：當清單中某個標的因代碼輸入錯誤、API 暫時失靈或數據缺失（如指數無淨值）而分析失敗時，系統會於終端捕捉異常並顯示錯誤標記（❌），但會繼續執行清單中其他標的的分析。
    * **價值**：這對於 CLI 工具的批次處理體驗至關重要，能確保使用者在一次操作中獲得最大化的資訊回饋，而非因單一錯誤導致整個自動化任務中斷。

## 4. 向下相容性實作細節

* **指令別名與參數轉發**：
    * v2.0 延續 `argparse` 的 `nargs="+"` 接收方式。
    * **自動降級邏輯**：當 `Engine` 判斷輸入僅為單一標位時，會自動切換為 v1.0 的垂直詳細報告模式；僅在偵測到多標的（或關鍵字 `watchlist`）時才啟動新版的比較表格。 使用者在操作習慣上完全無縫銜接。
* **非破壞性升級**：
    * `history.json` 採自動初始化設計，若檔案不存在則自動建立，完全不影響舊有使用者的查詢功能。

## 7. 已知限制與未來改進方向

* **限制**：目前批量抓取採用同步請求，當 `watchlist` 標的過多時，等待時間較長。
* **v3.0 規劃**：
    1. **非同步 IO**：引入 `aiohttp` 實現多標的同時併發抓取。
    2. **SQLite 遷移**：當數據量達到規模後，將 JSON 遷移至 SQLite 以提升查詢效率與數據關聯性。

## 8. Web Dashboard

v2.0 附帶一個輕量 Flask 儀表板，在瀏覽器中呈現 watchlist 的即時資料與歷史走勢圖。

### 啟動方式

```bash
cd v2/
pip install -r requirements.txt
python web_app.py
```

啟動後開啟瀏覽器前往 `http://localhost:5001`。

### 功能

- **Watchlist 總覽表**：自動讀取 `watchlist.txt`，顯示每檔標的的市價、淨值、折溢價（色碼 badge）、殖利率、量能比
- **歷史走勢圖**：點擊任一列，下方顯示該標的的本地歷史價格折線圖（資料來源：`data/history.json`）
- **重新整理**：右上角按鈕重新向 yfinance 抓取即時資料（約 10–20 秒）

### API 端點

| 路由 | 說明 |
| :--- | :--- |
| `GET /` | 回傳 Dashboard HTML |
| `GET /api/watchlist` | 回傳 watchlist 所有標的的即時資料（JSON） |
| `GET /api/history/<ticker>` | 回傳指定標的的本地歷史紀錄（JSON） |
